1.3 Flujo de Datos Detallado

Flujo de Pedido (Usuario)

  1. Navegación del Menú: El usuario accede a la página principal → Next.js renderiza el componente del menú → Se hace una petición GET a /api/products → El servidor consulta Railway PostgreSQL → Las imágenes se cargan desde Cloudinary CDN → Los datos se cachean con SWR
  2. Agregar al Carrito: Usuario hace clic en "Agregar" → El estado se actualiza localmente con React useState → Se valida disponibilidad del producto → Se calcula el subtotal
  3. Confirmar Pedido: Usuario hace clic en "Confirmar Pedido" → Se valida el carrito en el cliente → Se envía POST a /api/orders → Server Action valida datos con Zod → Se verifica autenticación con Supabase → Se crea registro en Railway PostgreSQL → Se retorna número de orden
  4. Pago: Usuario es redirigido a Mercado Pago → Completa el pago → Mercado Pago envía webhook → Se actualiza estado del pedido en Railway → Se envía notificación al usuario

Flujo de Administración

  1. Login Admin: Admin ingresa credenciales → POST a /api/auth/login → Supabase Auth valida → Se genera JWT token → Se almacena en cookie segura
  2. Crear Producto: Admin completa formulario → Sube imagen → Imagen se envía a Cloudinary → Cloudinary retorna URL optimizada → POST a /api/products con URL → Se crea registro en Railway PostgreSQL → UI se actualiza con SWR revalidation
  3. Gestionar Pedidos: Admin accede al panel → GET a /api/orders con filtros → Railway retorna pedidos → Se muestran en tiempo real → Admin actualiza estado → PUT a /api/orders/[id] → Se actualiza en Railway → Cliente recibe actualización

2.1 Endpoints de Productos

GET /api/products

Descripción: Obtiene la lista de todos los productos disponibles.

Query Parameters:

Parámetro Tipo Requerido Descripción
category string No Filtrar por categoría (ej: "bebidas", "comidas")
available boolean No Filtrar por disponibilidad (true/false)
search string No Buscar por nombre o descripción

Respuesta Exitosa (200):

{ "success": true, "data": [ { "id": "1", "name": "Café Americano", "description": "Café negro tradicional", "price": 3500, "image_url": "https://res.cloudinary.com/...", "category_id": "cafes", "available": true, "created_at": "2025-01-15T10:30:00Z" } ], "count": 150 }

Ejemplo de Uso:

// JavaScript/TypeScript const response = await fetch('/api/products?category=bebidas'); const data = await response.json(); console.log(data.data); // Array de productos

GET /api/products/[id]

Descripción: Obtiene los detalles de un producto específico.

Path Parameters:

Parámetro Tipo Descripción
id string ID único del producto

Respuesta Exitosa (200):

{ "success": true, "data": { "id": "1", "name": "Café Americano", "description": "Café negro tradicional", "price": 3500, "image_url": "https://res.cloudinary.com/...", "category": { "id": "cafes", "name": "Cafés" }, "available": true } }

Error (404):

{ "success": false, "error": "Producto no encontrado" }

POST /api/products

Descripción: Crea un nuevo producto (requiere rol admin).

Autenticación: Requerida (Admin)

Request Body:

{ "name": "Café Latte", "description": "Café con leche espumosa", "price": 4500, "image_url": "https://res.cloudinary.com/...", "category_id": "cafes", "available": true }

Validación (Zod Schema):

const productSchema = z.object({ name: z.string().min(3, "Mínimo 3 caracteres"), description: z.string().min(10), price: z.number().positive("Precio debe ser positivo"), image_url: z.string().url("URL inválida"), category_id: z.string(), available: z.boolean().default(true) });

Respuesta Exitosa (201):

{ "success": true, "data": { "id": "152", "name": "Café Latte", "created_at": "2025-01-28T14:20:00Z" }, "message": "Producto creado exitosamente" }

PUT /api/products/[id]

Descripción: Actualiza un producto existente (requiere rol admin).

Autenticación: Requerida (Admin)

Request Body (campos opcionales):

{ "name": "Café Latte Grande", "price": 5000, "available": false }

DELETE /api/products/[id]

Descripción: Elimina un producto (requiere rol admin).

Autenticación: Requerida (Admin)

Respuesta Exitosa (200):

{ "success": true, "message": "Producto eliminado exitosamente" }

2.2 Endpoints de Pedidos

GET /api/orders

Descripción: Obtiene los pedidos del usuario autenticado.

Autenticación: Requerida

Query Parameters:

Parámetro Tipo Descripción
status string Filtrar por estado: pending, preparing, ready, completed, cancelled
limit number Número de resultados (default: 20)
offset number Paginación (default: 0)

Respuesta Exitosa (200):

{ "success": true, "data": [ { "id": "ORD-2025-001", "user_id": "user123", "total": 15000, "status": "preparing", "pickup_time": "2025-01-28T12:30:00Z", "payment_method": "mercado_pago", "items": [ { "product_id": "1", "product_name": "Café Americano", "quantity": 2, "price": 3500, "subtotal": 7000 } ], "created_at": "2025-01-28T11:45:00Z" } ], "count": 15 }

POST /api/orders

Descripción: Crea un nuevo pedido.

Autenticación: Requerida

Request Body:

{ "items": [ { "product_id": "1", "quantity": 2 }, { "product_id": "5", "quantity": 1 } ], "pickup_time": "2025-01-28T12:30:00Z", "payment_method": "mercado_pago" }

Validaciones:

  • Todos los productos deben existir y estar disponibles
  • La hora de recogida debe ser al menos 15 minutos en el futuro
  • La hora de recogida debe estar en horario de operación (7:00 AM - 6:00 PM)
  • Las cantidades deben ser números positivos

Respuesta Exitosa (201):

{ "success": true, "data": { "order_id": "ORD-2025-156", "total": 15000, "pickup_time": "2025-01-28T12:30:00Z", "payment_url": "https://mercadopago.com/checkout/..." }, "message": "Pedido creado exitosamente" }

PUT /api/orders/[id]

Descripción: Actualiza el estado de un pedido (solo admin).

Autenticación: Requerida (Admin)

Request Body:

{ "status": "ready" }

Estados Válidos:

  • pending - Pedido recibido, pago pendiente
  • preparing - En preparación
  • ready - Listo para recoger
  • completed - Entregado
  • cancelled - Cancelado

3.4 Consultas SQL Comunes

Obtener productos con categoría:

SELECT p.id, p.name, p.description, p.price, p.image_url, p.available, c.name AS category_name FROM products p INNER JOIN categories c ON p.category_id = c.id WHERE p.available = true ORDER BY c.name, p.name;

Obtener pedidos con detalles:

SELECT o.id, o.total, o.status, o.pickup_time, o.created_at, u.full_name, u.email, COUNT(oi.id) AS items_count FROM orders o INNER JOIN users u ON o.user_id = u.id LEFT JOIN order_items oi ON o.id = oi.order_id WHERE o.status = 'preparing' GROUP BY o.id, u.full_name, u.email ORDER BY o.pickup_time ASC;

Productos más vendidos:

SELECT p.id, p.name, p.image_url, SUM(oi.quantity) AS total_sold, SUM(oi.subtotal) AS total_revenue FROM products p INNER JOIN order_items oi ON p.id = oi.product_id INNER JOIN orders o ON oi.order_id = o.id WHERE o.status = 'completed' AND o.created_at >= NOW() - INTERVAL '30 days' GROUP BY p.id, p.name, p.image_url ORDER BY total_sold DESC LIMIT 10;

4. SEGURIDAD

4.1 Autenticación y Autorización

Sistema de Autenticación

La plataforma utiliza Supabase Auth para gestionar la autenticación de usuarios:

  • JWT Tokens: Tokens firmados con algoritmo HS256
  • Duración de sesión: 7 días (configurable)
  • Refresh tokens: Renovación automática de tokens
  • Cookies seguras: HttpOnly, Secure, SameSite=Strict

Middleware de Autenticación

// middleware.ts import { createServerClient } from '@supabase/ssr'; export async function middleware(request) { const supabase = createServerClient( process.env.NEXT_PUBLIC_SUPABASE_URL, process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY ); // Verificar sesión const { data: { session } } = await supabase.auth.getSession(); // Proteger rutas admin if (request.nextUrl.pathname.startsWith('/admin')) { if (!session || session.user.role !== 'admin') { return NextResponse.redirect(new URL('/login', request.url)); } } return NextResponse.next(); }

Control de Acceso Basado en Roles (RBAC)

Rol Permisos
student • Ver menú
• Crear pedidos
• Ver sus propios pedidos
• Actualizar perfil
teacher • Todos los permisos de student
• Acceso prioritario en horarios pico
admin • Todos los permisos anteriores
• CRUD de productos
• Ver todos los pedidos
• Actualizar estado de pedidos
• Gestionar usuarios
• Acceso a dashboard y métricas

4.2 Protección de Datos

Encriptación

  • En tránsito: HTTPS/TLS 1.3 (certificado SSL de Vercel)
  • En reposo: Encriptación AES-256 en Railway PostgreSQL
  • Contraseñas: Hashing con bcrypt (cost factor: 12)
  • Tokens: JWT firmados con secreto de 256 bits

Row Level Security (RLS)

Políticas de seguridad a nivel de fila en PostgreSQL:

-- Los usuarios solo pueden ver sus propios pedidos CREATE POLICY "Users can view own orders" ON orders FOR SELECT USING (auth.uid() = user_id); -- Solo admins pueden actualizar productos CREATE POLICY "Only admins can update products" ON products FOR UPDATE USING ( EXISTS ( SELECT 1 FROM users WHERE id = auth.uid() AND role = 'admin' ) );

Sanitización de Inputs

  • XSS Prevention: Escape de HTML en todos los inputs de usuario
  • SQL Injection: Uso de prepared statements y ORM
  • CSRF Protection: Tokens CSRF en formularios

5. MANUALES DE USUARIO

5.1 Manual para Estudiantes

Paso 1: Registro e Inicio de Sesión

  1. Accede a la plataforma desde tu navegador: https://cafeteria-ucp.vercel.app
  2. Haz clic en "Registrarse" si es tu primera vez
  3. Ingresa tu correo institucional (@ucp.edu.co), nombre completo y contraseña
  4. Verifica tu correo electrónico (revisa spam si no llega)
  5. Inicia sesión con tus credenciales

Paso 2: Explorar el Menú

  1. En la página principal verás el menú completo con fotos de los productos
  2. Usa los filtros de categoría para encontrar lo que buscas: Bebidas, Comidas, Snacks, etc.
  3. Usa la barra de búsqueda para encontrar productos específicos
  4. Haz clic en un producto para ver su descripción completa y precio

Paso 3: Realizar un Pedido

  1. Haz clic en "Agregar al carrito" en los productos que desees
  2. Ajusta las cantidades usando los botones + y -
  3. Revisa tu carrito en la esquina superior derecha
  4. Haz clic en "Ver carrito" para revisar tu pedido
  5. Selecciona la hora de recogida (mínimo 15 minutos desde ahora)
  6. Verifica el total y haz clic en "Confirmar pedido"
  7. Serás redirigido a Mercado Pago para completar el pago

Paso 4: Seguimiento del Pedido

  1. Después del pago, recibirás un número de orden (ej: ORD-2025-156)
  2. Ve a "Mis Pedidos" en el menú de navegación
  3. Verás el estado de tu pedido en tiempo real:
    • Pendiente: Esperando confirmación de pago
    • En preparación: La cafetería está preparando tu pedido
    • Listo: Tu pedido está listo para recoger
  4. Cuando el estado cambie a "Listo", dirígete a la cafetería
  5. Muestra tu número de orden al personal
  6. Recoge tu pedido y disfruta

💡 Consejos:

  • Haz tu pedido al menos 20 minutos antes del receso para evitar esperas
  • Los horarios pico son 10:00 AM y 12:00 PM - planifica con anticipación
  • Puedes cancelar tu pedido hasta 5 minutos después de crearlo
  • Guarda tu número de orden por si necesitas consultar algo

7. DESPLIEGUE Y MANTENIMIENTO

7.1 Proceso de Despliegue

Despliegue en Vercel

  1. Conectar Repositorio:
    • Conecta tu repositorio de GitHub a Vercel
    • Vercel detecta automáticamente que es un proyecto Next.js
  2. Configurar Variables de Entorno:
    # Variables requeridas en Vercel DATABASE_URL=postgresql://... NEXT_PUBLIC_SUPABASE_URL=https://... NEXT_PUBLIC_SUPABASE_ANON_KEY=... CLOUDINARY_CLOUD_NAME=... CLOUDINARY_API_KEY=... CLOUDINARY_API_SECRET=... MERCADOPAGO_ACCESS_TOKEN=...
  3. Desplegar:
    • Vercel despliega automáticamente en cada push a main
    • Preview deployments para cada pull request
    • Rollback instantáneo si hay problemas

Configuración de Railway (Base de Datos)

  1. Crea un nuevo proyecto en Railway
  2. Agrega un servicio PostgreSQL
  3. Copia la DATABASE_URL y agrégala a Vercel
  4. Ejecuta las migraciones:
    npx prisma db push

CI/CD Pipeline

# .github/workflows/deploy.yml name: Deploy to Vercel on: push: branches: [main] jobs: deploy: runs-on: ubuntu-latest steps: - uses: actions/checkout@v2 - uses: actions/setup-node@v2 - run: npm install - run: npm run build - run: npm test - uses: amondnet/vercel-action@v20

7.4 Escalabilidad

Capacidad Actual

  • Usuarios concurrentes: Hasta 500 usuarios simultáneos
  • Pedidos por minuto: Hasta 50 pedidos/minuto
  • Almacenamiento de imágenes: 10GB en Cloudinary (plan gratuito)
  • Database connections: 100 conexiones simultáneas

Plan de Escalamiento

Si la demanda supera la capacidad actual:

  1. Fase 1 (100-200 pedidos/día): Configuración actual suficiente
  2. Fase 2 (200-500 pedidos/día):
    • Upgrade a Railway Pro ($20/mes)
    • Cloudinary Plus ($89/mes)
    • Implementar Redis para caché
  3. Fase 3 (500+ pedidos/día):
    • Database replication (read replicas)
    • CDN adicional para assets estáticos
    • Load balancing con múltiples instancias
